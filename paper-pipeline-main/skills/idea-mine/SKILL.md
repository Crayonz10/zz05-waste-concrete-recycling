## 核心使命

读取当前项目 `papers/` 目录下的一组学术论文PDF，通过"结构化精读 → idea 筛选评分 → idea-期刊匹配 → 主编适配 → 审稿人质控"五阶段 Pipeline，产出可直接灌入 `paper-init` 骨架的 idea 方案。

## 核心预设条件

**中文论文 = idea 池，不是已发表文献。** 在整个 pipeline 中，输入的中文/英文论文仅作为 idea 素材来源，不视为已发表的竞争文献。评估创新性时，参照系**只有英文文献**：这个 idea 在英文学术圈有没有人做过？对英文期刊审稿人来说是不是新的？中文论文中已发表的方法/发现不构成"已被做过"的判据。

## 学科锚点配置

### 亲和力四级制

| 等级 | 含义 | 是否进入学科推荐 |
|------|------|----------------|
| 🟢 自然适配 | idea 内核与期刊定位高度契合，只需常规叙事调整 | 是 |
| 🟡 可适配 | 需要补充建工/城市情境或调整叙事角度，但核心方法可复用 | 是 |
| 🔴 勉强 | 需要大幅改造才能勉强投稿，性价比低 | 否 |
| ⬛ 不适用 | 方法论或主题与期刊完全不兼容 | 否 |

**进入学科推荐的条件**：亲和力为 🟢 或 🟡 的期刊进入学科推荐，每个学科方向最多推荐 **3 本**（🟢 优先于 🟡，同级按适配度排序）。已在标准推荐中出现的期刊不重复推荐。

## 输出语言规范

Pipeline 全程生成的所有 md 文件（paper-notes、idea-raw、ideas、idea-review、final）**以中文为主要语言**。期刊名、方法论术语、理论名称、专有名词等可保留英文原文，但叙事、分析、评审意见、政策建议等正文内容一律使用中文。

## 项目目录规范

idea-mine 必须在符合以下标准结构的项目根目录下调用：

```
idea{NN}_{源论文研究领域}/
├── papers/                              ← 【输入】论文PDF池（必须存在）
│   ├── xxx.pdf
│   └── ...
├── paper-notes/                         ← 每篇论文的精读分析报告（自动生成，可缓存复用）
│   ├── xxx.md
│   └── ...
├── idea-raw/                            ← 【中间产物】阶段二筛选清单+评分 & 阶段三匹配结果（自动生成）
│   ├── _step1_candidates.md
│   ├── _step2_scores.md
│   ├── _step3_matching.md
│   └── ...
├── ideas/                               ← 【输出】最终 idea 方案（自动生成，每期刊独立文件）
│   ├── idea_01_{主题简称}_{期刊缩写}.md
│   └── ...
├── idea-review/                         ← 【中间产物】阶段五审稿人评审结果（自动生成）
│   ├── review_{期刊缩写}_P{NN}.md       ← 第一轮：每 idea × 每期刊的独立评审
│   ├── review_{期刊缩写}.md             ← 第二轮：主 agent 汇总的期刊评审报告（含横向排名）
│   └── ...
└── final_{YYYY-MM-DD}.md                ← 【输出】Pipeline 闭环总结（基本信息+排名表）
```

**命名说明**：
- **项目文件夹名**中的`{源论文研究领域}`描述的是输入论文池的研究主题（如"碳交易+可持续管理"）
- 论文统一放在 `papers/` 子目录，不要散落在项目根目录

## 前置检查

1. 检查 `papers/` 子目录是否存在，不存在则停止并提示用户：`请先创建 papers/ 目录并放入论文PDF`
2. 扫描 `papers/` 子目录中的 **PDF 文件**（`papers/**/*.pdf`）
3. 统计 PDF 数量并列出文件清单
4. 如果 PDF 数量为 0，停止并提示用户放入论文
5. 根据 PDF 数量给出建议：
   - **< 10 篇**：警告 idea 候选空间不足，建议补充至 12 篇以上，尤其注意方法论多样性
   - **10-20 篇**：甜蜜点（15篇左右最优），直接运行
   - **21-25 篇**：可以运行，提示"论文较多，注意力可能摊薄"
   - **> 25 篇**：建议用户分两批运行（每批 12-18 篇），或自行预筛至 20 篇以内。如用户坚持一次跑完，则执行但警告质量可能下降
6. 向用户确认：`发现 N 篇论文。开始挖掘？`

## Pipeline 执行

严格按以下五个阶段顺序执行。

---

### 阶段一：结构化精读（并行 subagents）

**目标**：提取每篇论文的可迁移idea内核。

**缓存机制**：精读结果存储在项目根目录的 `paper-notes/` 下，每篇论文一个 md 文件（文件名 = PDF 文件名去掉 `.pdf` 后缀）。论文内容不会变，**只要 md 存在就直接复用，不需要重新精读**。

对每篇 PDF，先检查 `paper-notes/{论文名}.md` 是否存在：
- 存在 → 直接读取，跳过精读
- 不存在 → 纳入精读队列

每篇 PDF 启动 **1 个独立的 Agent**（subagent_type: general-purpose, model: sonnet），1 篇 / 1 agent，避免单 agent 任务过重导致超时或失败时多篇连坐。采用**槽位制并发控制**：同时运行的 agent 数**不超过 8 个**，每当 1 个 agent 完成（或失败重试后仍失败），立即从队列中取下一篇启动，直到所有论文处理完毕。不等整批完成再启动下一批。每个 Agent 的 prompt 如下：

```
你是一名管理科学与工程方向的高级研究助理。请全文精读以下PDF论文，对每篇论文提取以下信息：

1. **研究情境/行业背景**：这篇论文研究的是什么行业/领域的什么问题？（1-2句话）
2. **一句话核心idea**：剥离具体行业情境后，这篇论文的可迁移学术贡献是什么？
3. **理论框架**：使用了什么理论？（如Weitzman框架、Hotelling模型、委托代理等）
4. **方法论标签**：博弈论/演化博弈/实证DID/面板回归/优化/预测/综述/DSGE/微分博弈/...
5. **核心机制标签**：倒U型/溢出效应/信息不对称/政策协同/委托代理/竞合博弈/...
6. **模型/方法的关键假设**：逐条列出每个核心假设（博弈论的参与者/策略/支付假设、实证的识别策略/排他性约束、优化的目标函数/约束条件等），并分析每个假设在迁移后是否仍然成立
7. **关键发现**（3条以内，每条展开到3-5句，必须包含具体的变量关系、效应大小/方向、非线性形式、边界条件等细节）
8. **核心创新点**（2-3个，每个展开说明：创新在哪里、为什么是创新、对迁移有什么价值。区分理论创新、方法创新、发现创新）
9. **数据来源与样本特征**（如有）：用了什么数据、样本量多大、时间跨度、数据库来源。纯理论建模论文标注"纯理论建模，无实证数据"即可

关键发现、核心创新点、关键假设三项尤其要充分展开，这些是后续 idea 迁移的核心素材。请逐字逐句仔细阅读，不要遗漏任何重要细节。

将每篇论文的精读结果直接写入 paper-notes/{论文名去掉.pdf后缀}.md，格式见下方模板。如果 paper-notes/ 目录不存在则先创建。

论文列表（路径来自前置检查扫描 papers/ 的结果，均带 papers/ 前缀）：
[论文路径列表]
```

**精读结果写入**：每篇论文的精读结果写入 `paper-notes/{论文名}.md`，格式：

```markdown
---
paper: {论文完整文件名}
analyzed: {YYYY-MM-DD}
---

## 1. 研究情境/行业背景
{内容}

## 2. 一句话核心 idea
{内容}

## 3. 理论框架
{内容}

## 4. 方法论标签
{内容}

## 5. 核心机制标签
{内容}

## 6. 模型/方法的关键假设
{逐条列出，分析迁移后是否成立}

## 7. 关键发现
{每条3-5句，含具体变量关系、效应、边界条件}

## 8. 核心创新点
{每个展开说明创新在哪、为什么、迁移价值}

## 9. 数据来源与样本特征（如有）
{内容，纯理论建模标注"纯理论建模，无实证数据"}
```

**失败处理**：如果某个 subagent 返回错误（如 API 连接失败），自动重试一次。重试仍失败则跳过该篇（该论文的 paper-notes/ md 不生成），提醒用户。Pipeline 继续执行（已成功精读的论文仍参与后续阶段）。

**汇总**：`paper-notes/` 下的所有 md 文件（含缓存复用的和新精读的）即为**论文摘要表**，供后续阶段直接读取使用。

**向用户展示方法论分布统计**，让用户确认论文池的方法论覆盖度：

```markdown
## 论文池方法论分布

| 方法论类型 | 篇数 | 论文 |
|-----------|------|------|
| 博弈论/机制设计 | N | 论文A, 论文B, ... |
| 实证因果推断(DID/IV) | N | ... |
| 演化博弈/仿真 | N | ... |
| 优化/预测 | N | ... |
| 综述/框架 | N | ... |
| 其他 | N | ... |
```

仅展示分布统计，不做判断性提醒（用户可能故意选同类方法论文来做创新点互补）。

---

### 阶段二：Idea 生成（识别可迁移内核）

**目标**：从论文池中识别具有迁移价值的 idea 内核，经评分排序后全部进入阶段三（idea-期刊匹配）。

> **执行方式**：Step 1-3 全部由主 agent 执行，不使用 subagent。不使用 WebSearch 做前置检索——直接**依赖模型知识**判断各 idea 在英文学术圈的研究密度和新颖性。

#### Step 1：逐篇迁移价值评估（主 agent 执行）

逐篇扫描 `paper-notes/` 下的精读报告，判断每篇论文的核心 idea 是否具备**独立的迁移价值**。

判断标准：
- idea 内核在英文学术圈确实未被充分探索（**参照系仅为英文文献**，中文论文是 idea 池不构成"已被做过"）
- 中文→英文跨语言迁移不存在抄袭问题，不需要担心这一点
- 只需评判：**idea 内核的学术贡献是否有迁移价值**、**方法论迁移后是否可行**（具体期刊适配在阶段四由主编处理，此步不评）

**硬筛选规则**（直接标记为不进入 Step 2，无需评估）：
- 综述类论文（如方法论综述、框架综述）→ 标记为 📚 综述
- 以 DID（双重差分）或 RDD（断点回归）为核心方法的论文 → 标记为 ❌，理由："DID/RDD 方法不采用"

产出：**主 idea 候选清单**，每篇标注状态和理由，写入 `idea-raw/_step1_candidates.md`：

```markdown
| # | 论文 | 状态 | 理由 |
|---|------|------|------|
| P1 | {论文名} | ✅ 值得迁移 | {简述迁移价值} |
| P2 | {论文名} | ❌ 不值得迁移 | {简述原因} |
| P3 | {论文名} | ⚠️ 存疑 | {简述疑虑} |
| P4 | {论文名} | 📚 综述 | {不作为独立idea源} |
```

**各状态在 Step 2 中的处理**：
- ✅ 值得迁移 → 进入 Step 2 评分
- ⚠️ 存疑 → **也进入 Step 2**（通过评分来量化——评分会体现在排名表中供阶段三期刊匹配参考，不需要提前排除）
- ❌ 不值得迁移 → 不进入 Step 2
- 📚 综述 → 不进入 Step 2

#### Step 2：逐篇评分（主 agent 直接执行）

**设计原则**：不构建中间"通用迁移 idea"——直接基于 `paper-notes/` 精读报告对原始 idea 的迁移价值打分。具体的期刊特定迁移在阶段四由主编从头完成，此处只做排序。

主 agent 逐篇扫描 Step 1 中标注为 ✅ 或 ⚠️ 的论文的精读报告，对每篇直接打分。

##### 评分维度

> **注意**：此为 Pipeline 内部的快速初筛评分（3维度/25分制），用于排序展示和供用户在阶段三确认期刊时参考（用户可根据评分决定是否缩减低分 idea 的期刊数）。与阶段五审稿人的五维度精评（1-10分/80分制）是完全独立的两套评分体系——初筛在前，精评在后，互不影响。

| 维度 | 分值 | 权重 | 说明 |
|------|------|------|------|
| 创新性 | 1-5 | ×2 | 该 idea 内核在英文学术圈有多新（参照系仅为英文文献） |
| 贡献 | 1-5 | ×2 | 对领域知识体系的理论/实践推进潜力 |
| 落地可行性 | 1-5 | ×1 | 方法论迁移后能否落地（数据可得性、核心假设是否成立） |

**总分** = 创新性×2 + 贡献×2 + 落地可行性×1，满分 25

##### 迁移创新性判断标准

所有 idea 都涉及迁移，统一适用以下判断（**参照系仅为英文文献**，中文论文是 idea 池不构成"已被做过"）：
- idea 内核在英文学术圈确实未被充分探索（以英文文献为准）
- **情境差异能产生与原论文不同的理论预测** → 创新性升级
- **理论预测方向与原论文一致** → 创新性降级（但迁移中文→英文本身有价值，体现在评分中）
- 有足够的数据/案例支撑落地（具体领域的数据可行性在阶段四由主编评估）

##### 产出

将所有评分结果写入 `idea-raw/_step2_scores.md`：

```markdown
| # | 论文 | 创新性(/5) | 贡献(/5) | 可行性(/5) | 总分(/25) | 评分理由 |
|---|------|-----------|---------|-----------|----------|---------|
| P1 | {论文名} | {X} | {X} | {X} | {X} | {1-2句核心理由} |
| P3 | {论文名} | {X} | {X} | {X} | {X} | {1-2句核心理由} |
| ... | ... | ... | ... | ... | ... | ... |
```

#### Step 3：汇总排序（主 agent 执行）

基于 `idea-raw/_step2_scores.md` 的评分，执行以下**纯机械操作**：

1. 按总分降序排列，**所有通过 Step 1 筛选的 idea（✅ 和 ⚠️）全部进入阶段三**，不做淘汰（因为每个 idea 独立匹配期刊，工作量已可控）
2. 在对话框中展示排名表摘要（编号 + 论文名 + 一句话核心idea + 方法论标签 + 总分）
3. 为每个入选 idea 确定**主题简称**（3-5 个字，从精读报告的"一句话核心 idea"中提取关键词，如"碳标签信息精度""CSR审计倒U型""竞合博弈"等），在排名表中展示，后续阶段四文件命名使用
4. 展示排名表后**直接进入阶段三**（不在此处向用户确认，确认环节在阶段三匹配时一并完成）

**方法论选择的决策优先级**（供阶段四主编参考）：
1. **主论文方法**（最高优先级）→ 用户选中的论文代表用户认可其方法，idea 的主框架方法跟随主论文
2. **数据/模型可行性** → 该方法迁移后能否落地（数据可得性、建模条件）。如果可行性存疑，不换方法，而是在可行性维度扣分

---

### 阶段三：Idea-期刊匹配

**目标**：为每个通过筛选的 idea 独立匹配最适合的目标期刊，用户确认后执行期刊画像调研，为阶段四主编适配做准备。

> **核心逻辑**：不同 idea 的方法论和主题领域不同，适合的期刊天然不同。每个 idea 独立推荐期刊，而非所有 idea 共享同一组期刊。

#### Step 1：迁移预判 + 期刊推荐（并行 subagents，model: opus）

每个通过筛选的 idea 启动 **1 个独立的 Agent**（subagent_type: general-purpose, model: opus），1 个 idea / 1 agent。采用**槽位制并发控制**：同时运行的 agent 数**不超过 8 个**，完成一个补一个。

每个 subagent 的 prompt：

```
你是一名管理科学与工程方向的资深学者，熟悉以下 28 本英文期刊的学术定位、
读者群体、方法论偏好和选题风格。

你的任务：为 1 个待迁移的 idea 推荐 **5 本**最适合的目标期刊。

**关键要求**：你推荐期刊的依据是该 idea **迁移后**的适配度，而不是原论文
研究情境的适配度。迁移后的 idea 可能留在原领域，也可能落地到
完全不同的领域——你需要先判断最有价值的迁移方向，再匹配期刊。

**请用 Read 工具读取以下文件**：
1. 原始论文精读报告：{paper-notes/{论文名}.md 的完整绝对路径}
2. 期刊池列表：{~/.claude/skills/idea-mine/journals.md 的完整绝对路径}

**执行步骤**：

1. **迁移方向预判**：基于精读报告的方法论内核（§3-§4）和核心机制（§5），
   判断该 idea 迁移后最可能落地的 **2-3 个迁移方向**。每个方向需覆盖三层信息：
   - **落地场景**（1句）：迁移到什么领域/什么问题

对于学科推荐期刊，**优先参考学科迁移方向中的叙事调整要点**，确保叙事从建工/城市可持续的实践痛点切入，而非直接使用运筹味的叙事框架。
方法论跟随主论文（主论文方法最高优先级）。

**请将适配方案写入以下文件**：
{ideas/idea_{NN}_{主题简称}_{期刊缩写}.md 的完整绝对路径}
```

**每个适配版本的输出格式**（精简版 §1-§5，控制输出长度）：

```markdown
### Idea {N} — {期刊名} 适配版本

> **目标期刊**: {期刊名}

#### 1. 灵感来源 & 参考论文
- **主论文**：{论文名} — {一句话核心idea}
- **迁移方向**：{1-2句}
- **迁移关键差异**：{1-2句，迁移后与原论文产生了哪些不同的理论预测或机制}

#### 2. 行业背景与实践难题 → Gap/Objective/RQ

**行业背景与实践难题**：{3-4句，讲清楚迁移后的行业情境、实践痛点和故事线}

**文献不足**：{1-2句，现有研究为什么没解决这个问题}

| Gap | Objective | RQ |
|:----|:----------|:---|
| G1: {gap} | {objective} | RQ1: {rq} |
| G2: {gap} | {objective} | RQ2: {rq} |

#### 3. 方法论
{4-6句：跟随主论文的什么方法、原论文方法如何映射到新情境、核心假设迁移后是否成立、迁移后有哪些关键建模差异、数据从哪来/样本怎么构建/有没有现成数据库}

#### 4. 贡献意图
**理论贡献**：{1句}
**实践贡献**：{1句}

#### 5. 风险评估
TODO（阶段五审稿人 challenge 产出后回填）
```

**产出**：每个 idea × 其确认期刊 = 全部适配版本，以独立文件存储在 `ideas/` 目录下（`idea_{NN}_{主题简称}_{期刊缩写}.md`），进入阶段五审稿。

### 阶段五：审稿人质量控制（两轮拆分）

**目标**：以目标期刊审稿人身份对所有 idea 做严格评估，最终生成每期刊的横向排名。

#### 第一轮：逐 idea 独立评分（并行轻量 agent）

**实现方式**：每个 idea × 该 idea 的每个确认期刊 = **1 个独立的评分 Agent**（subagent_type: general-purpose, model: sonnet）。每个 agent 只评审 **1 个 idea** 的适配版本，读取量极小（3 个文件：期刊画像 + 适配版本 + 精读报告）。采用**槽位制并发控制**：同时运行的 agent 数**不超过 8 个**，完成一个补一个。共需启动 Σ(每个 idea 的确认期刊数) 个 agent。主 agent 读取 `idea-raw/_step3_matching.md` 确定每个 idea 对应哪些期刊。各 agent **互不知晓其他 idea 的评分结果**，确保独立性。

每个评分 agent 的人设根据阶段三的期刊画像**独立生成**。prompt 模板：

```
你现在是 {期刊名} 的资深审稿人（Associate Editor 级别）。你见过太多
"换个情境重做一遍"的投稿，对真正的创新有极高的辨别力。

重要预设：你评审的这个 idea 来源于一个非公开的 idea 池（不是已发表文献），
因此不存在"这个已经被做过"的问题。你只需从英文学术圈的视角判断这个 idea
是否有创新性和贡献。

作为该期刊的审稿人，你特别关注：
{根据该期刊画像动态生成 2-3 条特殊关注点}

**请用 Read 工具读取以下文件**：
1. 本刊画像：{_journal-spec/{期刊全称}.md 的完整绝对路径}
2. 适配版本：{ideas/idea_{NN}_{主题简称}_{期刊缩写}.md 的完整绝对路径}
3. 原始论文精读报告：{paper-notes/{论文名}.md 的完整绝对路径}

对这个 idea，你需要：
1. 对适配版本做五维度评分
2. 给出 1-2 个核心 challenge + 预防性回应建议

五维度评分 rubric（每项 1-10 分）：
| 维度 | 1-3分 | 4-6分 | 7-10分 |
| 创新性 | 简单换情境/拼凑 | 有一定新意但非突破 | 打开新的研究方向 |
| 贡献 | 对理论和实践都没有明确推进 | 拓展了现有理论框架或有一定实践指导意义 | 理论突破+实践价值兼备 |
| 方法可行性 | 数据不可得/模型不可解 | 有挑战但可克服 | 数据现成/方法成熟 |
| 时效性 | 过时话题 | 常规热点 | 踩中政策窗口/前沿热点 |
| 发表概率 | <10% | 30-50% | >70% |

> 创新性评分参照系仅为英文文献，中文论文作为 idea 池不构成"已被做过"的判据。

**请将评审结果写入以下文件**：
{idea-review/review_{期刊缩写}_P{NN}.md 的完整绝对路径}
```

**失败处理**：如果某个评分 agent 失败，自动重试一次。重试仍失败则跳过该 idea 在该期刊的评审，不阻塞 pipeline。

#### 单 idea 评审输出格式（精简，控制篇幅）

> **重要**：每个评审文件控制在 **300 字以内**。只输出评分表 + 每维度一句话理由 + challenge，不展开论述。

```markdown
# {期刊名} 审稿人评审 — Idea P{NN}: {主题简称}

| 维度 | 评分 | 评分理由（1句） |
|------|------|---------------|
| 创新性 | X/10 | {1句} |
| 贡献 | X/10 | {1句} |
| 方法可行性 | X/10 | {1句} |
| 时效性 | X/10 | {1句} |
| 发表概率 | X/10 | {1句} |

**Challenge**：{1-2句核心质疑}
**预防性回应**：{1-2句}
```

#### 第二轮：主 agent 汇总排名（无 subagent）

第一轮全部完成后，主 agent 执行以下操作：

1. 读取所有 `idea-review/review_{期刊缩写}_P{NN}.md` 文件
2. 按期刊分组，提取五维度评分，计算综合分（创新×2 + 贡献×2 + 可行×1.5 + 时效×1 + 发表×1.5，满分 80）
3. 在每个期刊内部按综合分降序排列，生成横向排名
4. 将所有单 idea 评审内容 + 横向排名表合并写入 `idea-review/review_{期刊缩写}.md`（期刊汇总文件）

期刊汇总文件格式：

```markdown
# {期刊名} 审稿人评审报告

（此处依次粘贴各 idea 的评审内容，来自 review_{期刊缩写}_P{NN}.md）

---

## 横向排名

| 排名 | Idea | 来源 | 综合分(/80) |
|------|------|------|------------|
| 1 | P{NN}: {主题简称} | 标准 | X |
| ... | ... | ... | ... |

> 综合分 = 创新×2 + 贡献×2 + 可行×1.5 + 时效×1 + 发表×1.5，满分 80。
> 来源列：标准 = 纯 idea 驱动推荐；建工🟢/🟡 或 城市🟢/🟡 = 学科亲和力推荐。
```

#### §5 风险评估回填

pipeline 完成后，用户选定 idea 时由主 agent 根据 challenge 直接回填适配版本的 §5：
- 每个 challenge 作为一行风险
- 等级根据审稿人判断标注（⚠️中 / 🔴高）
- 应对策略来自审稿人的预防性回应建议

## Pipeline 完成后的操作

用户可随时（当天或数周后）选择感兴趣的 idea 进行初始化：

```
用户：我要初始化 P{NN}，投 {期刊名}，项目路径 {path}，编号 {ID}
```

主 agent 执行：
1. 读取该 idea 在指定期刊的适配版本
2. 读取 review 文件中该 idea 的 challenge，回填 §5 风险评估
3. 按 paper-init 的 idea.md 模板生成完整 idea.md（§1-§5，§5 已含 challenge）
4. 根据期刊的 `@publisher` 标签从 journals.md 推断出版社
5. `mkdir -p {path}` → `cd {path}` → 运行 `/paper-init {ID}`
6. paper-init 成功后，将 idea.md 写入 `structure/0_global/idea.md`
7. **打包 idea-mine 上下文到新项目**（见下方详细说明）

#### Step 7：打包 idea-mine 上下文

将 idea-mine 项目中与该 idea 相关的所有上下文材料复制到新项目的 `structure/0_global/idea-context/` 目录，为后续 idea 细化、文献规划和方法讨论提供完整参考。

**目录结构**：

```
structure/0_global/
├── idea.md                        ← 正式的 idea 方案（§5 已回填 challenge）
└── idea-context/                  ← idea-mine 上下文包（本步骤生成）
    ├── source_paper.pdf           ← 原始论文 PDF（从 papers/ 复制并重命名）
    ├── paper_note.md              ← 精读报告（从 paper-notes/ 复制）
    ├── migration_analysis.md      ← 从 _step3_matching.md 提取该 idea 的迁移方向预判段落
    ├── adaptations/               ← 该 idea 的所有期刊适配版本
    │   ├── idea_{NN}_{主题}_{期刊1}.md
    │   ├── idea_{NN}_{主题}_{期刊2}.md
    │   └── ...
    └── reviews/                   ← 该 idea 的所有审稿评审
        ├── review_{期刊1}_P{NN}.md
        ├── review_{期刊2}_P{NN}.md
        └── ...
```

**具体操作**：

1. **创建目录**：`mkdir -p {path}/structure/0_global/idea-context/adaptations {path}/structure/0_global/idea-context/reviews`
2. **复制原始论文 PDF**：从 idea-mine 项目的 `papers/` 目录中找到该 idea 对应的论文 PDF，复制到 `idea-context/source_paper.pdf`。论文文件名从 `paper-notes/{论文名}.md` 的 frontmatter `paper` 字段获取
3. **复制精读报告**：将 `paper-notes/{论文名}.md` 复制到 `idea-context/paper_note.md`
4. **提取迁移方向预判**：从 `idea-raw/_step3_matching.md` 中提取该 idea 对应的迁移方向预判段落（包括匹配表中该行 + "## 迁移方向预判"中该 idea 的完整内容 + "## 学科迁移方向"中该 idea 的完整内容 + "## 学科亲和力评估"中该 idea 的完整内容），写入 `idea-context/migration_analysis.md`
5. **复制所有适配版本**：将 `ideas/idea_{NN}_*.md`（该 idea 编号的所有期刊适配文件）复制到 `idea-context/adaptations/`
6. **复制所有审稿评审**：将 `idea-review/review_*_P{NN}.md`（该 idea 编号的所有单篇评审文件）复制到 `idea-context/reviews/`

**用途说明**：这些上下文材料在新项目中的作用：
- `source_paper.pdf` + `paper_note.md`：随时回查原文方法细节和核心假设，支撑 idea 细化讨论
- `migration_analysis.md`：迁移方向的理论差异和假设变化，供 `/lit-plan` 设计检索方向时参考
- `adaptations/`：横向对比不同期刊的叙事策略差异，帮助打磨目标期刊版本
- `reviews/`：审稿人的 challenge 和预防性回应，指导方法论设计和风险规避

**到此为止**，后续的文献工作流（/lit-plan → /lit-review → /lit-tag → /lit-pool）由用户在项目目录中自行启动。

## 输出文件

- `final_{YYYY-MM-DD}.md` — Pipeline 闭环总结（基本信息 + 匹配表 + 排名表）
- `ideas/` — 最终 idea 方案（每个 idea × 其确认期刊）
- `idea-review/` — 第一轮单 idea 评审（`review_{期刊缩写}_P{NN}.md`）+ 第二轮期刊汇总（`review_{期刊缩写}.md`）
